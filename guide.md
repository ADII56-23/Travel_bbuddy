# Travel Buddy: Project Documentation & Viva Preparation Guide

This document provides a comprehensive overview of the **Travel Buddy** web application. Use this guide to easily understand the architecture, technologies used, data flow, and current state of the project for your viva.

## 1. Project Overview
Travel Buddy is an intelligent, full-stack travel recommendation platform. It aims to simplify travel planning by allowing users to search for destinations, define budgets, specify constraints (duration, origin), and receive curated travel packages. The application smartly blends **Machine Learning (Content-Based Filtering)** and **Generative AI** to ensure users always receive high-quality suggestions, even when exact database matches aren't available.

## 2. Tech Stack & Technologies Used

### Backend
* **Python**: The core programming language for the backend data processing and server logic.
* **Flask**: A lightweight Python web framework used to handle routing, HTTP requests, and rendering frontend templates.
* **Werkzeug**: Used for secure password hashing and verification (`generate_password_hash`, `check_password_hash`).

### Frontend
* **HTML & CSS**: Structuring and styling the user interfaces across various pages (Templates include `index`, `explore`, `saved_trips`, `activities`, etc.).
* **JavaScript**: Adds interactivity to the frontend (e.g., dynamically fetching recommendations, updating the DOM via AJAX/Fetch API without reloading the page).

### Database
* **SQLite (`sqlite3`)**: A local serverless relational database used for its simplicity and built-in Python support. Stored locally in `users.db`. 

### Machine Learning & Data Processing
* **Pandas**: Used internally for loading the dataset (`travel_packages.csv`) and doing heavy data-manipulation (filtering, sorting).
* **Scikit-Learn (sklearn)**: Used to generate **TF-IDF Vectors** (Term Frequency-Inverse Document Frequency) and calculate **Cosine Similarity** between what the user searches for and what the packages contain.
* **NLTK (Natural Language Toolkit)**: Used to preprocess the text data (lowercasing, lemmatization, and stop-word removal) so the ML model only looks at meaningful keywords.

### External APIs
* **OpenRouter API (Google Gemini 2.0 Flash)**: Acts as the backbone for the fallback recommendation engine, conversational Chatbot functionality, and complex dynamic requests like Itinerary generation and nearby activities lookups.
* **ip-api**: A public API used temporarily when a user visits to guess their current location, providing better default context for travel origin points.

## 3. Project Structure: How The Pieces Connect

The codebase is structured modularly. Here is the high-level connection tree:

### `web_app.py` (The Central Hub)
This is the main entry point of the server. 
1. **At Startup**: It automatically establishes the Database connection, runs `init_db()` to make sure the users table is ready, loads the pandas dataset via `data_loader.py`, and immediately trains/pre-processes the ML model so it's ready in memory.
2. **Routes**: It defines all frontend page routing (`/`, `/explore`, `/profile`) and API endpoints used by frontend JavaScript (`/api/recommend`, `/api/generate_itinerary`).
3. **Session Management**: It handles maintaining a logged-in state by putting a distinct `user_id` inside a secure Flask cookie session.

### `src/` (The Logic Engines)
* `data_loader.py`: A simple wrapper for pandas `read_csv`.
* `preprocessing.py`: Takes the dirty DataFrame, cleans out nulls, tokenizes strings, and runs the `TfidfVectorizer` to map text packages into mathematical numbers.
* `recommendation.py`: The actual matching engine. Filters packages by budget/destination constraints, then uses `cosine_similarity` to rank the remaining packages based on the user's "interests". 
* `location_utils.py`: Estimates budgets and distance depending on where the user currently is.

### `templates/` (The Views)
Holds the actual HTML layouts. The Flask backend uses `render_template` to serve these to the user. Included inside are Jinja tags (like `{{ current_user }}`) to inject dynamic backend variables into the frontend interface.

## 4. How the App Actually Works (Data Flow Example)

**Scenario: A user searches for a trip to Paris with a budget of ₹150,000 for 5 days.**

1. **Request Received**: The user frontend hits the `/api/recommend` POST endpoint.
2. **Origin Detection**: The server checks the logged-in user profile for their origin. If unauthenticated, it uses `session` context generated by querying the `ip-api`. 
3. **Budget Feasibility Check**: It checks `estimate_min_viable_budget()`. If ₹150,000 is physically impossible, it instantly returns an error message telling the user to increase their budget.
4. **Local Database Check**: It calls `filter_packages()` to look inside `travel_packages.csv` for Paris packages under ₹150k.
5. **Machine Learning Match**: If matches exist, it runs Cosine Similarity to find the *best* Paris match based on the user's specific written interests and applies an artificial "Similarity Score" for the UI.
6. **AI Fallback Execution**: If *no* packages are found in the CSV, the system does not break. Instead, it hits the **OpenRouter API** passing a strict prompt to act as the database and intelligently *generate* 5 JSON travel packages matching the criteria.
7. **Response to Client**: JSON is passed back to the `script.js` frontend which iterates over the results and renders "Cards" onto the user's screen.

## 5. What is Implemented VS What is Not

### ✅ What You Already Have (Implemented & Working)
* **Authentication System**: Full signup/login routing with secure password hashing (`werkzeug.security`).
* **Profile Management**: Profile updating, detecting user IP context when logged out vs logged in.
* **Hybrid Search Engine**: Working content-based ML recommendation system hooked directly to a brilliant LLM fallback system if local data fails.
* **Conversational AI Chatbot**: Users can request custom help from a System-Prompted "Travel Buddy API Agent" via OpenRouter.
* **Dynamic Content APIs**: Working endpoints that auto-generate activities (`/api/activities-search`) and full itinerary breakdowns (`/api/generate_itinerary`) dynamically for any destination in the world relying purely on prompt integration with OpenRouter.

### ❌ What is NOT Implemented / Future Scope (Good for Answering "Where to Improve")
* **Booking Integration**: Users can look at recommendations, but they cannot actually "purchase" or "book" the flights/hotels directly through a payment gateway (like Stripe or Razorpay). 
* **Live Flight/Hotel Pricing APIs**: The system estimates budgets and prices either mathematically or through LLM knowledge gaps. It does not hit live Amadeus or Skyscanner APIs for minute-by-minute accurate flight quotes.
* **Production Database**: SQLite is great for local prototyping, but in a real-world scenario with thousands of concurrent users writing data, this would need to migrate to **PostgreSQL** or **MySQL**.
* **Cloud Infrastructure & WSGI Servers**: The application currently runs on the default Flask development server (`app.run(debug=True)`). To deploy, it needs a production WSGI server like Gunicorn or uWSGI, and it will need to be deployed to a cloud provider like AWS, Vercel, or Heroku.
* **Complete Automated Tests**: Missing robust unit-testing (like PyTest suites) to check if API endpoints break during upgrades.
